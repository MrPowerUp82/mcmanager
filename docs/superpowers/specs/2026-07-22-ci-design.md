# CI Linux+Windows — Design

> Documento produzido via sessão de brainstorming em 2026-07-22.
> Status: **Aprovado**.
> Escopo: quinta e última parte da Fase 4 (ver `docs/design-melhorias.md`, seção 6). Triagem de vulnerabilidades, Dashboard, `mcmanager doctor` e erros estruturados já estão prontos.

---

## 1. Resumo de Entendimento

- **O quê:** workflow do GitHub Actions com matriz Linux + Windows para a suíte de testes, mais um job de lint com `ruff`. Corrige de quebra uma inconsistência real encontrada: `pyproject.toml` declara `requires-python = ">=3.8"`, mas o Django 5.1.15 já fixado exige Python >=3.10 de fato.
- **Por quê:** design original da Fase 4 previa isso desde o início ("GitHub Actions: matriz Linux + Windows, lint com ruff"); até agora não existe nenhuma automação de CI no projeto — toda verificação de teste/lint é manual, no ambiente de dev do mantenedor.
- **Para quem:** mantenedor solo, garante que mudanças futuras (via PR ou push direto) não quebrem silenciosamente em uma das duas plataformas suportadas.
- **Escala:** 1 workflow, 2 jobs, sem necessidade de infraestrutura própria (runners hospedados do GitHub).

### Suposições confirmadas

1. `requires-python` corrigido para `>=3.10` (alinha com o que o Django 5.1.15 já exige de fato — testar numa matriz que incluísse 3.8/3.9 falharia na instalação da dependência, não no código do projeto).
2. Matriz de Python: uma única versão, `3.14` (a mesma usada localmente pelo mantedor) — não uma matriz de múltiplas versões de Python. Suporte a múltiplas versões de Python pode ser considerado no futuro, se necessário.
3. `ruff` adicionado como nova dev dependency (o projeto hoje usa `autopep8`/`djlint`, mas só como formatadores manuais, sem gate de CI) — roda `ruff check .` com configuração mínima (sem customizar o conjunto de regras além do padrão do próprio ruff, que já é conservador).
4. CI também roda `npm ci && npm run build:css`, para pegar automaticamente regressões como a do caminho quebrado do `build:css` corrigida recentemente (Fase 4, parte 1).
5. Risco conhecido e aceito: como o `ruff` nunca rodou neste código antes, a primeira execução real pode revelar violações pré-existentes — resolvido durante a implementação (corrigidas ou o conjunto de regras ajustado), não adivinhado no design.

---

## 2. Abordagem escolhida

Um único arquivo de workflow (`.github/workflows/ci.yml`) com 2 jobs independentes:
- `lint`: roda uma vez (só `ubuntu-latest`, já que resultado de lint independe de plataforma), `ruff check .`.
- `test`: matriz `[ubuntu-latest, windows-latest]`, suíte pytest completa + build do CSS.

Nenhuma infraestrutura própria, nenhum runner self-hosted — usa exclusivamente GitHub-hosted runners, adequado à escala do projeto (mantedor solo, sem necessidade de builds pesados/paralelos além do que os runners padrão já oferecem).

## 3. Design

### `pyproject.toml`

```diff
-requires-python = ">=3.8"
+requires-python = ">=3.10"
```

Adiciona uma seção mínima de configuração do ruff:

```toml
[tool.ruff]
target-version = "py310"
```

(Sem customizar `select`/`ignore` além do padrão do próprio ruff — ajustado durante a implementação se a primeira execução real revelar problemas.)

### `requirements-dev.txt`

Adiciona `ruff` à lista de dependências de desenvolvimento (junto de `pytest`/`pytest-django`).

### `.github/workflows/ci.yml`

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.14'
      - run: pip install ruff
      - run: ruff check .

  test:
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, windows-latest]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.14'
      - run: pip install -r requirements.txt -r requirements-dev.txt
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm run build:css
      - run: pytest
```

Pontos importantes:
- `fail-fast: false` na matriz de teste — uma falha no Windows não cancela a execução do Linux (e vice-versa), permitindo ver o resultado das duas plataformas mesmo se uma quebrar.
- `lint` e `test` são jobs independentes, rodam em paralelo — lint não bloqueia a matriz de teste, e vice-versa.
- `pull_request` sem filtro de branch — roda em qualquer PR contra qualquer branch-alvo (comportamento padrão do GitHub Actions).

### Testes

Não há "testes" no sentido tradicional para este trabalho — a verificação é a própria CI rodando com sucesso. A implementação deve confirmar, antes de considerar a tarefa concluída:
1. `ruff check .` roda localmente (se possível neste ambiente) ou documentar que só foi possível verificar via CI real, e reportar quantas violações (se houver) precisaram de correção.
2. A suíte pytest completa continua passando após a mudança de `requires-python` (não deve haver nenhuma mudança de comportamento de código, só de metadado de empacotamento).
3. `npm run build:css` continua funcionando (já verificado localmente na Fase 4, parte 1 — este é só sobre automatizar a checagem, não mudar o comportamento).
4. O workflow em si só pode ser verificado de fato rodando no GitHub após o push — a implementação deve deixar isso explícito no relatório final, já que não há como simular um runner do GitHub Actions localmente com certeza total.

## 4. Decision Log

| # | Decisão | Alternativas consideradas | Justificativa |
|---|---------|---------------------------|---------------|
| 1 | Corrigir `requires-python` para `>=3.10` | Deixar como está (`>=3.8`) | Aprovado pelo usuário — alinha a declaração com o que já é verdade na prática (Django 5.1 exige 3.10+) |
| 2 | Matriz de Python: só 3.14 | Matriz com 3.10 (mínimo) + 3.14 (mais recente) | Aprovado pelo usuário — mantedor solo, sem pressão real de suportar múltiplas versões de Python ainda; simplicidade sobre abrangência |
| 3 | Adicionar `ruff` como lint gate novo | Usar `autopep8`/`djlint` já existentes como gate de CI | Aprovado pelo usuário — fecha o item explícito do design original ("lint com ruff"); `autopep8`/`djlint` continuam existindo como formatadores manuais, sem mudança |
| 4 | Incluir `npm run build:css` na CI | Só testes Python | Aprovado pelo usuário — pega automaticamente o tipo de regressão de caminho quebrado já corrigida manualmente na Fase 4, parte 1 |
| 5 | 2 jobs separados (`lint` + `test` em matriz), rodando em paralelo | 1 job só, lint + teste sequencial por combinação de SO | Lint é independente de SO — rodar separado evita repetir a mesma checagem 2x (uma por SO) sem necessidade |
| 6 | `fail-fast: false` na matriz de teste | Deixar o padrão (`fail-fast: true`) | Permite ver o resultado de ambas as plataformas mesmo se uma falhar — mais útil para diagnosticar problemas específicos de uma plataforma |
