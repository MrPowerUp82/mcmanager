# CI Linux+Windows Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a GitHub Actions workflow that runs the test suite on a Linux+Windows matrix and lints the codebase with `ruff`, and fix a stale `requires-python` floor that no longer matches the project's actual Django dependency.

**Architecture:** One workflow file, two independent jobs — `lint` (runs once, `ubuntu-latest`) and `test` (matrix `[ubuntu-latest, windows-latest]`). `pyproject.toml` gets a corrected `requires-python` and a minimal `[tool.ruff]` section; `requirements-dev.txt` gains `ruff`.

**Tech Stack:** GitHub Actions (`actions/checkout`, `actions/setup-python`, `actions/setup-node`), `ruff` (new dev dependency), no other new tooling.

## Global Constraints

- `requires-python` becomes exactly `>=3.10` (Django 5.1.15's actual floor) — was `>=3.8`, stale since the Django version bump.
- CI's Python version is `3.14` only — no multi-version Python matrix.
- `ruff` config in `pyproject.toml` is minimal (`target-version = "py310"` only) — do not add custom `select`/`ignore` rules unless real violations are found and need to be scoped down; if ruff reports violations against its own default rule set, prefer fixing the actual code over suppressing the rule, unless the volume makes that impractical (use judgment, document the decision either way).
- The `lint` job runs only on `ubuntu-latest` (lint results don't depend on OS) — do not add it to the test matrix.
- The `test` matrix uses `fail-fast: false` so a failure on one OS doesn't cancel the other.
- Workflow triggers: `push` to `main` and `pull_request` (no branch filter on `pull_request`).
- **Known environment limitation**: this project's dev machine has a Windows "App Control" policy that blocks the `ruff` binary from executing locally (confirmed during design — `pip install ruff` succeeds, but `ruff` itself fails with `OSError: [WinError 4551] Uma política de Controle de Aplicativo bloqueou este arquivo`). If you hit this exact error when trying to run `ruff` locally, do NOT attempt to bypass or work around a security policy — this is expected on this machine. Document that local verification wasn't possible and that verification will happen for real once the workflow runs on GitHub's own runners (which don't have this restriction). Do not skip attempting it first, though — this policy is specific to this one machine's configuration, and it's possible it doesn't block it in the environment you're actually running in.

3. **It fails with `OSError: [WinError 4551] Uma política de Controle de Aplicativo bloqueou este arquivo`** (or a similarly-worded Windows App Control block) — this is a known, expected limitation of this specific dev machine, not a bug in your work. Do not attempt to bypass it. Document that local verification wasn't possible for this reason, and that real verification will happen once Task 2's workflow runs on GitHub's own runners.

### Task 2: GitHub Actions workflow

**Files:**
- Create: `.github/workflows/ci.yml`

**Interfaces:**
- Consumes: `ruff` and its `[tool.ruff]` config (Task 1) for the `lint` job; `requirements.txt`/`requirements-dev.txt` (existing) and `npm run build:css` (existing, from the dependency-vulnerability-triage plan's fix) for the `test` job.
- Produces: nothing consumed by later tasks — this is the final task in this plan.

- [ ] **Step 1: Create the workflow directory and file**

Create `.github/workflows/ci.yml`:

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.14'
      - run: pip install ruff
      - run: ruff check .

  test:
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, windows-latest]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.14'
      - run: pip install -r requirements.txt -r requirements-dev.txt
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm run build:css
      - run: pytest
```

- [ ] **Step 2: Validate the YAML is well-formed**

Run:
```bash
python -c "import yaml; yaml.safe_load(open('.github/workflows/ci.yml', encoding='utf-8'))" && echo "Valid YAML"
```
Expected: `Valid YAML` printed, no exception. (If `yaml` isn't importable, run `pip install pyyaml` first — it's a transitive dependency of other tools in this environment already in most cases, but install it explicitly if missing; this is a one-off local check, not a new project dependency, so do not add it to `requirements-dev.txt`.)

- [ ] **Step 3: Commit**

```bash
git add .github/workflows/ci.yml
git commit -m "ci: add GitHub Actions workflow with Linux+Windows test matrix and ruff lint"
```
